## Problem: cutoff_len=2048 截断大量 CoT 序列

### 显存限制

单卡 24GB 显存（24564 MiB），使用 4-bit QLoRA + Flash Attention 2 + gradient checkpointing 后，最大 `cutoff_len` 只能设为 **2048 tokens**。尝试更大值时（如 4096、8192、16384）会遇到 CUDA OOM。

smoke test 配置（`qwen3_8b_lora_cot_pro858_smoke.yaml`）中设置了 `cutoff_len: 16384`，但仅限 64 条样本且未成功完成完整训练。

### 截断影响

| cutoff_len | 被截断样本数 | 占比 |
|-----------|------------|------|
| 2048 | 522 / 858 | **60.8%** |
| 4096 | 281 / 858 | 32.8% |
| 8192 | 38 / 858 | 4.4% |
| 16384 | 1 / 858 | 0.1% |

在 `cutoff_len=2048` 下，**60.8% 的样本被截断**。截断发生在序列尾部，即 **response（CoT 推理 + KG JSON）被部分丢失**。

具体来说：
- 对于 median 样本（total 2,572 tokens，prompt 1,071，response 1,480）：在 2048 截断时，response 丢失约 524 tokens（约 35% 的 response 被截掉）
- 对于 P90 样本（total 6,943 tokens，prompt 3,102，response 3,946）：response 被完全截断，且 prompt 前半部也被截掉
- 长尾样本（max 20,983 tokens）：超过 90% 的内容被截断

这意味着模型在训练时**对大多数样本只学到了部分 CoT 推理链**，尤其是对较长的病历文本（user text P90=2,318 tokens），模型从未见过对应的完整 KG 抽取结果。

## Solution 1: 截断 Prompt 侧，为 Response 留空间

### 思路

当前 `cutoff_len=2048` 对 prompt + response 统一截断。由于 system prompt 固定 771 tokens（全部样本相同），user text 又存在长尾（P95=2,756，max=15,350），可以在数据预处理阶段截断过长的 user text，将有限的 token 预算更多分配给 response。

### 具体方案

在 `convert_to_llamafactory.py` 中新增 `--max_user_tokens` 参数，对 user content 做截断：

1. **保留完整 system prompt**（771 tokens）—— 固定的 KG schema 定义
2. **截断 user text**：设置上限（如 500 tokens），超出部分从头部保留 + 尾部截断
3. **保留更多 response**：截断后 prompt 控制在 771 + 500 + chat template overhead ≈ **1,300 tokens**，留给 response 约 **748 tokens**

### 效果估算

假设 prompt 截断后固定 ~1,300 tokens，不同 cutoff_len 下的可用空间：

| cutoff_len | 可用于 Response | 能覆盖的 Response 分位数 |
|-----------|----------------|----------------------|
| 2048 | ~748 tokens | ~P20（748 / 1,480 median） |
| 4096 | ~2,796 tokens | ~P70（接近 P75=3,028） |
| 8192 | ~6,892 tokens | ~P99+ |

关键瓶颈仍然是 cutoff_len 本身——**仅靠截断 prompt 无法在 2048 下覆盖 median response（1,480 tokens）**。但如果与更高 cutoff_len 配合（如 4096，需要解决显存问题），效果会显著改善。

### 实现方式

```python
# convert_to_llamafactory.py 中新增参数
parser.add_argument("--max_user_tokens", type=int, default=None)

# 对 user content 应用 tokenizer 截断
if max_user_tokens is not None:
    user_ids = tokenizer.encode(user_content)
    if len(user_ids) > max_user_tokens:
        user_ids = user_ids[:max_user_tokens]
        user_content = tokenizer.decode(user_ids)
```

### 局限性

- 在 cutoff=2048 下效果有限，必须配合提高 cutoff_len 才有效
- 截断 user text 对极长病历文本（max 15,350 tokens，约 1% 样本）会丢失关键临床信息
- 这部分长尾样本更适合用 `chunk_merge` 模式（分块抽取 + 合并）在数据生成阶段处理

## Solution 2: 切换至 Qwen3-4B

### 思路

Qwen3-4B 参数量约为 Qwen3-8B 的一半，在 4-bit QLoRA 下显存占用显著降低。节省的显存可用于提高 `cutoff_len`。

### 显存估算

| Component | Qwen3-8B (4-bit) | Qwen3-4B (4-bit) |
|-----------|-----------------|-----------------|
| Base Model Weights | ~4 GB | ~2 GB |
| LoRA Adapter (rank=8) | ~84 MB | ~42 MB |
| KV Cache + Activations (cutoff=2048) | ~12 GB | ~6 GB |
| KV Cache + Activations (cutoff=4096) | OOM (~24 GB+) | ~12 GB |
| KV Cache + Activations (cutoff=8192) | OOM | ~24 GB（临界） |

注：以上为粗略估算，实际显存受 flash_attention（O(1) KV cache）、gradient_checkpointing（减少 activations）、batch_size 等因素影响。

### 目标 cutoff 分析

基于上述 token 分布数据：

| cutoff_len | 覆盖样本率 | 对 4B 预期可行 |
|-----------|-----------|--------------|
| 2048 | 39.2% | 是（保守） |
| 4096 | 67.2% | 是（预期） |
| 8192 | 95.6% | 临界 / 需要验证 |
| 16384 | 99.9% | 可能 OOM |

理想目标是 **cutoff_len=8192**，可覆盖 95.6% 的样本。若 8192 仍然 OOM，4096 也可覆盖 67.2%（比 8B @ 2048 的 39.2% 多覆盖 28% 的样本）。

### 优势

- 可大幅提高 cutoff_len，覆盖更多完整 CoT 序列
- 训练和推理速度更快
- 可复用现有的 LLaMA-Factory pipeline 和 QLoRA 配置

### 劣势

- 4B 模型的 KG 抽取质量可能低于 8B，需要对比实验验证
- 需确认 Qwen3-4B base model 在 Huawei HF mirror 上的可用性
