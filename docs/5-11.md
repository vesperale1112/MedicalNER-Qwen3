# Qwen3 医学信息抽取模型微调 OOM 问题进展汇报

## 1. 当前实验背景

目前我正在使用 LLaMA Factory 对 Qwen3 系列模型进行医学信息抽取任务的监督微调。当前任务主要是让模型根据输入文本，结合给定的医学知识图谱 schema，输出结构化抽取结果。训练方式为 LoRA SFT，底座模型最初选择的是 Qwen3-8B，显卡环境为单张 NVIDIA A10，显存约 24GB。

当前主要配置如下：

```yaml
model_name_or_path: /hpc_stor03/sjtu_home/xiran.wang/models/Qwen3-8B
quantization_bit: 4
quantization_method: bitsandbytes

stage: sft
finetuning_type: lora
lora_rank: 8
lora_target: all

cutoff_len: 8192
per_device_train_batch_size: 1
gradient_accumulation_steps: 8
bf16: true
gradient_checkpointing: true
```

从日志来看，实际训练时已经使用了 4bit 量化、LoRA、batch size 1、gradient checkpointing 等常见节省显存策略，但在 `cutoff_len=8192` 时仍然发生 OOM。日志中显示本次训练使用单张 A10，显存约 24564MiB，训练配置中 `cutoff_len` 为 8192，且已经开启 `gradient_checkpointing: true`。

---

## 2. 当前遇到的问题

### 2.1 OOM 现象

训练在刚开始阶段发生 CUDA OOM，关键报错为：

```text
torch.OutOfMemoryError: CUDA out of memory.
Tried to allocate 4.19 GiB.
GPU 0 has a total capacity of 23.56 GiB,
of which 3.31 GiB is free.
Including non-PyTorch memory, this process has 20.24 GiB memory in use.
Of the allocated memory 19.87 GiB is allocated by PyTorch.
```

更具体地，报错发生在 causal LM loss 计算阶段：

```text
loss = self.loss_function(logits=logits, labels=labels, ...)
logits = logits.float()
```

也就是说，OOM 不只是模型加载阶段的问题，而是在 forward / loss 计算时，模型需要额外分配一个约 4GB 的张量。日志中也显示 Qwen3-8B 的词表大小为 `vocab_size: 151936`，hidden size 为 4096，层数为 36 层。

### 2.2 初步判断

我目前的判断是：
**这次 OOM 的主因不是模型权重本身，而是长上下文训练时 activation、logits 和 loss 计算带来的显存峰值。**

对于 causal LM SFT，最后一层 logits 的形状大致是：

```text
batch_size × sequence_length × vocab_size
```

在当前设置下：

```text
batch_size = 1
sequence_length = 8192
vocab_size ≈ 151936
```

这个 logits 张量本身就可能达到数 GB 级别。如果后续 loss 计算中发生 `logits.float()`，显存峰值会进一步上升。因此，即使模型已经使用 4bit 量化和 LoRA，长上下文训练仍然可能超出 A10 24GB 的承载能力。

---

## 3. 已完成的尝试

### 3.1 降低 cutoff_len

我将 `cutoff_len` 从 8192 降到 2048 后，训练可以正常开始运行，说明 OOM 的确与序列长度高度相关。

但是，从数据长度统计来看，2048 虽然能跑，但会截断大量样本：

```text
num samples: 858
min: 1133
p50: 2572
p75: 5463
p90: 6943
p95: 7946
p99: 9852
max: 20983

>2048: 522
>4096: 281
>8192: 38
>16384: 1
```

这说明：

| cutoff_len | 被截断样本数 | 截断比例 | 覆盖率 |
| ---------: | -----------: | -------: | -----: |
|       2048 |    522 / 858 |    60.8% |  39.2% |
|       4096 |    281 / 858 |    32.8% |  67.2% |
|       8192 |     38 / 858 |     4.4% |  95.6% |
|      16384 |      1 / 858 |     0.1% |  99.9% |

因此，`cutoff_len=2048` 只能作为调试配置，不太适合作为最终训练配置。
如果使用 2048，超过 60% 的样本会被截断，可能导致模型学习到不完整的 reasoning 或不完整的 JSON 输出。

---

## 4. 当前分析：问题可能来自三部分

### 4.1 Prompt / schema 过长

目前每条训练样本中都包含较完整的医学知识图谱 schema、实体类型、关系类型和输出规则。根据训练日志中的样本预览，system prompt 中确实包含大量 schema 信息和规则说明。

这会带来两个问题：

1. 每条样本都重复相同的 schema，浪费大量 token；
2. 真正与当前输入文本相关的信息比例下降；
3. 在 `cutoff_len` 较小时，assistant 的答案部分可能被截断，影响监督信号质量。

因此，后续需要重点考虑是否可以压缩 prompt，例如：

* 删除重复解释；
* 将完整 schema 改为更紧凑的实体 / 关系列表；
* 将长描述改为短标签；
* 去除非必要的 CoT 引导；
* 只保留输出格式约束和最关键的抽取规则。

### 4.2 Qwen3-8B 对单张 A10 来说偏重

虽然当前已经使用了 4bit 量化和 LoRA，但 Qwen3-8B 在 8192 上下文长度下仍然显存压力较大。尤其是 Qwen3 的词表较大，loss 计算时 logits 张量开销明显。

这意味着：
**如果继续使用单张 A10，Qwen3-8B 可能只能稳定支持 2048 或 3072，4096 需要进一步优化后尝试，8192 较困难。**

### 4.3 普通多卡不一定解决单卡 OOM

多卡训练需要区分两种情况：

1. 普通 DDP；
2. DeepSpeed ZeRO / FSDP 等参数或 optimizer state 分片训练。

如果只是普通 DDP，每张 GPU 仍然会各自保留一份完整模型，并各自处理一个完整样本，因此不能从根本上解决单卡长序列 OOM。
如果使用 DeepSpeed ZeRO-3 或 FSDP，则可以把参数、梯度、optimizer state 分散到多张卡上，有机会缓解显存压力。

不过，这次 OOM 有一部分来自长序列 logits / loss 计算，这部分不一定能被 ZeRO 完全切分。因此，多卡可以改善，但不能保证直接解决 8192 长度训练。

## 5. 后续计划

目前我考虑同时从三个方向推进。

---

### 方案一：修剪数据与 prompt

这是我认为最应该优先做的方向。因为当前数据中 prompt 和 schema 很长，即使换模型或加显卡，也会继续浪费大量上下文。

计划：

1. 统计 system、user、assistant 三部分分别占多少 token；
2. 判断被截断的主要是 prompt 还是 assistant 输出；
3. 压缩 system prompt；
4. 重新统计长度分布；
5. 比较压缩前后在 2048 / 4096 / 8192 下的样本覆盖率。

目标是尽量将 p50 和 p75 压到 4096 以下，使 Qwen3-8B + 4096 成为可行方案。

需要讨论的问题：

* 当前完整 schema 是否必须在每条样本中重复出现？
* 是否可以只保留实体类型和关系类型，而删除长解释？
* 是否必须训练 CoT？还是可以只训练最终结构化输出？
* 如果保留 CoT，是否可以压缩 reasoning 模板？

---

### 方案二：改用 Qwen3-4B

如果 prompt 压缩后，Qwen3-8B 仍然只能在 2048 或 3072 下稳定运行，我考虑改用 Qwen3-4B。

优势：

1. 显存压力明显降低；
2. 更有希望使用 4096 或 8192 的 `cutoff_len`；
3. 对结构化抽取任务来说，完整监督样本可能比更大的参数规模更重要。

我的判断是：

```text
Qwen3-4B + cutoff_len 4096/8192
可能优于
Qwen3-8B + cutoff_len 2048
```

原因是后者虽然模型更大，但训练样本中大量内容会被截断，尤其可能破坏 JSON 输出格式和完整推理链条。

需要讨论的问题：

* 当前任务是否强依赖 8B 的复杂推理能力？
* 医学实体关系抽取任务中，4B 是否已经足够？
* 是否可以先用 4B 跑通长上下文版本，作为 baseline？

---

### 方案三：启用多卡训练

如果组里资源允许，我希望尝试多卡训练，尤其是 DeepSpeed ZeRO-3，而不是普通 DDP。

计划测试顺序：

1. 先用 2 卡测试多卡环境是否正常；
2. 使用 Qwen3-8B + ZeRO-3 + cutoff_len 4096；
3. 如果 4096 稳定，再尝试 6144；
4. 如果资源允许，再尝试 8192。

双卡时可以考虑如下设置：

```yaml
cutoff_len: 4096
per_device_train_batch_size: 1
gradient_accumulation_steps: 4
gradient_checkpointing: true
deepspeed: examples/deepspeed/ds_z3_config.json
```

这里将 `gradient_accumulation_steps` 从 8 改为 4，是为了在双卡时保持原来的有效 batch size：

```text
单卡: 1 × 8 × 1 = 8
双卡: 1 × 4 × 2 = 8
```

需要讨论的问题：

* 组里是否可以提供 2 张或 4 张 A10 / A100 / H 系列显卡？
* 是否允许使用 DeepSpeed ZeRO-3？
* 如果多卡资源有限，是否优先保证 Qwen3-8B，还是改用 Qwen3-4B 跑更长上下文？

---

## 6. 推荐的实验优先级

我目前建议按下面顺序推进：

### Step 1：先修剪 prompt

目标是降低样本长度，减少无效 token 重复。

预期收益：

* 直接降低显存；
* 减少截断；
* 提高训练信号密度；
* 对 8B 和 4B 都有帮助。

---

### Step 2：尝试 Qwen3-8B + cutoff_len 4096

在 prompt 压缩后，重新尝试：

```yaml
model_name_or_path: Qwen3-8B
cutoff_len: 4096
per_device_train_batch_size: 1
gradient_accumulation_steps: 8
gradient_checkpointing: true
bf16: true
eval_strategy: no
```

如果可以跑通，再打开 eval。

---

### Step 3：如果 8B + 4096 仍然不稳定，切换 Qwen3-4B

尝试：

```yaml
model_name_or_path: Qwen3-4B
cutoff_len: 4096
```

如果显存允许，再尝试：

```yaml
cutoff_len: 8192
```

---

### Step 4：如果需要保留 Qwen3-8B 和长上下文，申请多卡 + ZeRO-3

多卡不是第一优先级，但如果组里资源允许，可以尝试：

```yaml
model_name_or_path: Qwen3-8B
cutoff_len: 4096 / 6144 / 8192
deepspeed: examples/deepspeed/ds_z3_config.json
```

---

## 7. 当前需要组会上讨论的问题

1. **schema 是否必须完整放进每条训练样本？**
   如果不是，是否可以改成更短的 schema summary？
2. **是否必须保留 CoT？**
   如果最终任务只需要结构化 JSON，是否可以训练模型直接输出结果，而不是完整 reasoning？
3. **模型大小和上下文长度哪个更优先？**
   当前选择是：
   * Qwen3-8B + 短上下文；
   * Qwen3-4B + 长上下文。
4. **是否可以使用多卡资源？**
   如果可以，希望优先尝试 ZeRO-3，而不是普通 DDP。
5. **最终评估标准是什么？**
   是更关注实体识别 F1、关系抽取 F1，还是 JSON 格式稳定性？
   这会影响是否保留 CoT、是否压缩输出、是否过滤长样本。

---

## 8. 当前结论

目前 OOM 的主要原因不是单纯的模型参数过大，而是 Qwen3-8B 在长上下文 SFT 时，由 activation、logits 和 loss 计算共同造成的显存峰值。`cutoff_len=2048` 已经可以运行，说明降低序列长度是有效的，但从数据长度分布看，2048 会截断超过 60% 的样本，因此不适合作为最终方案。

后续我计划优先压缩 prompt 和数据格式，然后尝试在 Qwen3-8B 上使用 4096 长度训练。如果仍然 OOM，则考虑切换到 Qwen3-4B，以换取更长上下文和更完整的监督信号。如果组里可以提供多卡资源，则进一步尝试 DeepSpeed ZeRO-3，以支持 Qwen3-8B 的 4096 或更长上下文训练。
